Indexing & Query Optimization

Indexing is used to speed up the retrieval of records from a database. Without an index, the DBMS must perform a "Full Table Scan."

How it works (B-Trees):

Most databases use a B-Tree structure. Think of it like the index at the back of a textbook; it tells the system exactly which page (or data block) to look at.

-- Creating an index on the 'email' column
CREATE INDEX idx_student_email ON Students(email);
The Cost of Indexing: While indexes make SELECT queries faster, they make INSERT and UPDATE slower because the index must be updated every time data changes!

Transaction & Concurrency Control

A transaction is a unit of work that must be treated as a single operation (ACID properties).

Concurrency Problems:

  • Dirty Read: Reading data that hasn't been "committed" yet.
  • Lost Update: Two users update the same record at once, and one overwrites the other.

Deadlocks:

A situation where Transaction A waits for a resource held by B, and B waits for a resource held by A. Neither can move forward.

The Solution: Locking

  • Shared Lock (S): Allows multiple transactions to read but not write.
  • Exclusive Lock (X): Only one transaction can access the data (no one else can even read it).

Database Security & Backup

Data is the most valuable asset of a company. It must be protected from loss and theft.

Security Strategies

  • Authentication: Using strong passwords and Multi-Factor Auth.
  • Authorization: Using the "Principle of Least Privilege" (Granting only the access needed).
  • Encryption: Scrambling data at rest (on the disk) and in transit (over the network).

Backup Strategies

  • Full Backup: Copying the entire database.
  • Incremental: Only backing up data that changed since the *last* backup.
  • Differential: Only backing up data that changed since the last *full* backup.
