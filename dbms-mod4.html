The CAP Theorem

In a distributed system (multiple servers), it is impossible to simultaneously provide more than two out of the following three guarantees:

[Image of CAP theorem diagram showing Consistency Availability and Partition Tolerance]
ConsistencyEvery user sees the same data at the same time, no matter which server they connect to.
AvailabilityThe system stays up and running, responding to requests even if some servers fail.
Partition ToleranceThe system continues to operate even if communication between servers is broken.

The Trade-off: On the Internet, "Partition Tolerance" is mandatory (networks fail), so you must choose between CP (Consistency) or AP (Availability).

NoSQL Data Models

NoSQL databases don't use tables. Instead, they use flexible structures based on the use case.

1. Document Store (e.g., MongoDB)

Stores data in JSON-like documents. Highly flexible; every record can have different fields.

{
  "id": 101,
  "name": "John",
  "tags": ["developer", "python"]
}

2. Key-Value Store (e.g., Redis)

The simplest type. Data is stored as a unique key and its associated value. Extremely fast for caching and session management.

3. Column-Family (e.g., Cassandra)

Optimized for reading/writing massive amounts of data across many servers (used by Facebook and Netflix).

When to choose NoSQL over SQL?

Feature SQL (Relational) NoSQL (Non-Relational)
Schema Strict & Pre-defined Flexible & Dynamic
Scaling Vertical (Bigger Server) Horizontal (More Servers)
Best For Complex Joins, Accounting Social Media, Real-time Data
Integrity Strong (ACID) Sacrificed for Speed (BASE)

The "BASE" Philosophy:

Unlike ACID, NoSQL often follows BASE: Basically Available, Soft-state, Eventual consistency.
