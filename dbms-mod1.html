DBMS vs File Systems

Before DBMS, data was stored in simple files (like Excel or Text files). This led to major issues that DBMS solved.

Key Differences:

  • Redundancy: File systems often have duplicate data; DBMS minimizes it.
  • Inconsistency: In file systems, updating a phone number in one file might not update it in another. DBMS ensures data is consistent.
  • Security: DBMS offers granular access control (who can see what), which is hard to manage in basic files.
  • Concurrent Access: Multiple users can work on a DBMS at once without corrupting data.

ACID Properties

For a database transaction to be considered reliable, it must follow the ACID rules.

AAtomicity
All or Nothing. A transaction either fully completes or fails entirely.
CConsistency
The database must move from one valid state to another valid state.
IIsolation
Transactions happening at the same time don't interfere with each other.
DDurability
Once a transaction is committed, it stays saved even if the system crashes.
Real-World Example: Think of a Bank Transfer. If money leaves your account (Step 1) but the system crashes before it hits the receiver's account (Step 2), Atomicity ensures the money is returned to you.

Three-Schema Architecture

This architecture is designed to hide the complexity of physical storage from the user.

  • 1. External Level (View Level): What the user sees (e.g., a profile page). Users only see the parts of the data relevant to them.
  • 2. Conceptual Level (Logical Level): Describes what data is stored and the relationships between them (Tables and Columns).
  • 3. Internal Level (Physical Level): How the data is actually stored on the hard drive (blocks, bits, etc.).

Data Independence

This allows you to change the physical storage (moving from HDD to SSD) without changing the way the user views the data.
