Entity-Relationship (ER) Diagrams

ER Diagrams are visual blueprints used to represent the logical structure of a database.

Core Components:

  • Entity (Rectangle): A real-world object (e.g., Student, Teacher, Course).
  • Attribute (Oval): Properties of an entity (e.g., Student Name, Roll No).
  • Relationship (Diamond): How entities interact (e.g., Student *Enrolls* in Course).

Cardinality (Relationships):

  • 1:1 (One-to-One): A person has one Passport.
  • 1:N (One-to-Many): A Department has many Employees.
  • M:N (Many-to-Many): Many Students enroll in many Courses.

Normalization: Reducing Redundancy

Normalization is the process of organizing data to minimize duplication and prevent "Anomalies" (Update, Delete, and Insert errors).

  • 1NF (First Normal Form): Eliminate repeating groups. Every cell must contain a single (atomic) value.
  • 2NF (Second Normal Form): Must be in 1NF + Remove partial functional dependencies (all non-key attributes must depend on the whole primary key).
  • 3NF (Third Normal Form): Must be in 2NF + Remove transitive dependencies (non-key attributes should not depend on other non-key attributes).
  • BCNF (Boyce-Codd Normal Form): A stricter version of 3NF for complex keys.
Memory Hook: For 3NF, remember: "The data must depend on the key, the whole key, and nothing but the key, so help me Codd."

Database Keys

Keys are used to uniquely identify rows and establish relationships between tables.

Primary KeyUniquely identifies every record in a table. Cannot be NULL.
Foreign KeyA field in one table that links to the Primary Key of another table.
Candidate KeyA set of columns that *could* be the primary key.
Composite KeyA primary key that consists of more than one column.
Referential Integrity: This is the rule that a Foreign Key must always point to a valid Primary Key in another table, preventing "orphan" records.
